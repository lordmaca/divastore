## 1. Why this contract exists

Today, Divinha (DivaHub's AI agent) handles customer conversations on Instagram,
Shopee, and Mercado Livre. We want the **same** Divinha to answer questions,
recommend products, and build a cart inside the Brilho de Diva storefront at
`https://loja.brilhodediva.com.br`.

To do that, DivaHub needs to expose a small **public HTTP API** that the
storefront can call from its BFF layer on every chat turn. This document is the
minimum contract the storefront needs to implement its chat UI against.

**What already exists (for reference):**

- `divahub/lib/integrations/conversations/pipeline.ts` ŌĆö webhook ŌåÆ persist ŌåÆ AI turn
- `divahub/lib/integrations/conversations/ai.ts` ŌĆö LLM orchestration, product search, playbooks
- `divahub/lib/integrations/conversations/cart.ts` ŌĆö cart snapshot + storefront deep-link builder
- `divahub/app/admin/conversations/page.tsx` ŌĆö admin inbox (session-authenticated, ADMIN role)

**What is missing:** a thin public HTTP surface that lets the storefront invoke
the same pipeline for a browser-initiated conversation.

## 3. Endpoints

### 3.1 `POST /api/public/divinha/turn` ŌĆö primary

Runs one conversation turn. **Streaming (SSE) is required**; a non-streaming
fallback is defined in ┬¦8.

**Request headers:**

| Header                  | Required | Notes                                               |
| ----------------------- | -------- | --------------------------------------------------- |
| `Authorization`         | yes      | `Bearer <token>` ŌĆö see ┬¦4                           |
| `Content-Type`          | yes      | `application/json`                                  |
| `Accept`                | yes      | `text/event-stream` for SSE, `application/json` for fallback |
| `X-Brilho-Request-Id`   | yes      | UUID generated by storefront, echoed in every SSE event and in logs |
| `X-Brilho-Version`      | yes      | `1` ŌĆö contract major version                        |
| `Idempotency-Key`       | optional | If present, DivaHub must dedupe for 10 min          |

**Request body:**

```jsonc
{
  "conversationId": "cnv_01H...",      // nullable on first turn; DivaHub mints if absent
  "channel": "storefront_web",         // enum: storefront_web (future: storefront_mobile)
  "locale": "pt-BR",                   // always pt-BR for v1
  "user": {
    "customerId": "c_01H...",          // nullable (guest)
    "sessionKey": "bd_..." ,           // storefront session id; always present
    "email": "cliente@example.com",    // nullable; only if logged in
    "firstName": "Camila",             // nullable
    "isAuthenticated": true
  },
  "message": {
    "id": "msg_01H...",                // storefront-generated ULID
    "role": "user",
    "content": "Me sugere um presente pra minha m├Że at├® R$ 150",
    "attachments": []                  // reserved ŌĆö always empty in v1
  },
  "cartSnapshot": {
    "items": [
      { "variantId": "v_123", "sku": "BD-ANEL-17", "name": "Anel Solit├Īrio", "qty": 1, "priceCents": 12900 }
    ],
    "subtotalCents": 12900,
    "couponCode": null,
    "currency": "BRL"
  },
  "context": {
    "referrerPath": "/loja/aneis",     // where in the site the user opened the chat
    "viewedProductSlug": null,         // slug of the current PDP, if any
    "utmSource": "home_hero",          // nullable
    "deviceHint": "mobile"             // "mobile" | "desktop" | "tablet"
  },
  "history": [                         // last N turns ONLY on recovery (normally empty; DivaHub owns history)
    { "role": "assistant", "content": "..." }
  ]
}
```

**Response ŌĆö SSE stream (`Content-Type: text/event-stream`):**

Each event has an `event:` type and `data:` JSON. Types:

- `turn.start` ŌĆö once, immediately. `{ "turnId": "...", "conversationId": "..." }`
- `token` ŌĆö zero or more. `{ "delta": "Ol├Ī" }` ŌĆö streaming text deltas for the assistant reply.
- `message` ŌĆö zero or more. Fully-formed assistant message (used for non-delta payloads like product cards). See ┬¦6.
- `action` ŌĆö zero or more. Structured side-effect the storefront must execute. See ┬¦7.
- `error` ŌĆö on failure. `{ "code": "llm_unavailable", "message": "...", "retryable": true }`
- `turn.end` ŌĆö once, last. `{ "turnId": "...", "totalTokens": 123, "stopReason": "end" }`

Example:

```
event: turn.start
data: {"turnId":"t_01H...","conversationId":"cnv_01H..."}

event: token
data: {"delta":"Perfeito!"}

event: token
data: {"delta":" Separei tr├¬s op├¦├Ąes:"}

event: message
data: {"id":"m_01H...","role":"assistant","kind":"product_carousel","products":[{"slug":"anel-coracao-rosa","reason":"dentro do or├¦amento"}]}

event: action
data: {"type":"show_product","slug":"anel-coracao-rosa"}

event: turn.end
data: {"turnId":"t_01H...","totalTokens":412,"stopReason":"end"}
```

**Response ŌĆö non-streaming fallback (`Accept: application/json`):**

```jsonc
{
  "turnId": "t_01H...",
  "conversationId": "cnv_01H...",
  "messages": [ /* array of Message ŌĆö ┬¦6 */ ],
  "actions":  [ /* array of Action  ŌĆö ┬¦7 */ ],
  "stopReason": "end"
}
```

**Status codes:**

| Code | Meaning                                                   |
| ---- | --------------------------------------------------------- |
| 200  | Streaming or JSON response delivered                      |
| 400  | Malformed body                                            |
| 401  | Missing/invalid bearer token                              |
| 403  | Token valid but not authorized for this channel           |
| 409  | `Idempotency-Key` already used with a different body      |
| 422  | Schema validation failed ŌĆö include `issues: ZodIssue[]`   |
| 429  | Rate-limited. Include `Retry-After` header (seconds)      |
| 503  | LLM/upstream dependency down. Retryable.                  |

### 3.3 `POST /api/public/divinha/conversations/:id/handoff` ŌĆö optional (v1.1)

Marks a conversation for human handoff. Storefront calls this when the customer
clicks "falar com atendente" or Divinha emits a `handoff_human` action and the
customer confirms. Body: `{ reason: "customer_request" | "divinha_escalated", note?: string }`.

**v1 can skip this.** The `handoff_human` action alone is enough for the
storefront to surface a banner; DivaHub admin inbox becomes the source of truth.

## 4. Authentication

Mirror the inbound pattern already in use at
[lib/integration/divahub/auth.ts](./auth.ts) (just reversed):

- **Transport:** `Authorization: Bearer <token>`
- **Token issuance:** DivaHub mints a long-lived API key per calling service.
  Issue one for the storefront; the storefront stores it as an encrypted secret
  (`divahub.apiKey`) and sends it on every request.
- **Storage on DivaHub side:** SHA-256 hash only. Never keep plaintext in the DB.
- **Comparison:** constant-time (`crypto.timingSafeEqual`) against the stored hash.
- **Rotation:** support two active keys simultaneously (current + next) so
  rotation is zero-downtime. Admin UI at `divahub/app/admin/integrations/...`
  (re-use the pattern from the storefront's rotating-key UI).
- **Audit:** every authenticated request writes a row to DivaHub's integration
  log with `{ keyHint, ip, turnId, latencyMs, status }`.

**Do not** re-use Divinha's Instagram webhook secret ŌĆö different threat model,
different rotation cadence. Issue a dedicated key.

## 5. Rate limits

DivaHub must enforce per-key token-bucket limits. Recommended defaults
(mirroring the storefront's inbound budget):

- **Chat turns:** 120 req/min per storefront session (`sessionKey`), burst 20.
- **Per API key (storefront aggregate):** 600 req/min, burst 60.
- **Health:** 60 req/min per IP.

On 429, return JSON `{ "error": "rate_limited", "retryAfterMs": 4200 }` and the
`Retry-After` header in seconds.

The storefront BFF will layer its own per-session rate limit
([lib/rate-limit.ts](../../rate-limit.ts)) before calling DivaHub, so most abuse
is stopped at the edge. DivaHub's limit is defense in depth.

## 6. Message shape

```ts
type Message =
  | { id: string; role: "assistant"; kind: "text"; content: string }
  | { id: string; role: "assistant"; kind: "product_carousel"; products: ProductRef[] }
  | { id: string; role: "assistant"; kind: "quick_replies"; prompts: string[] }
  | { id: string; role: "assistant"; kind: "cart_preview"; items: CartItemRef[] }
  | { id: string; role: "assistant"; kind: "link"; label: string; href: string };

type ProductRef = {
  slug: string;          // canonical storefront slug ŌĆö storefront hydrates price + image
  variantSku?: string;   // optional ŌĆö specific variant recommendation
  reason?: string;       // short "why this one" caption to render under the card
};

type CartItemRef = {
  variantId: string;
  qty: number;
};
```

**Critical:** `ProductRef` carries **no price, no image URL, no stock**.
Divinha must **never** quote prices or claim stock ŌĆö the storefront hydrates
those at render time from its own DB. This is non-negotiable for regulatory and
consumer-protection reasons (C├│digo de Defesa do Consumidor, art. 30 ŌĆö informa├¦├Żo
vincula o fornecedor).

## 7. Action taxonomy

Actions are structured side-effects the storefront will execute on the user's
behalf. Every action must be deterministic, idempotent, and traceable. Any
action not in this list must be rejected by the storefront.

```ts
type Action =
  | { type: "add_to_cart"; variantSku: string; qty: number; reason?: string }
  | { type: "remove_from_cart"; variantId: string }
  | { type: "update_cart_qty"; variantId: string; qty: number }
  | { type: "apply_coupon"; code: string }
  | { type: "remove_coupon" }
  | { type: "show_product"; slug: string }
  | { type: "navigate"; path: string }                    // whitelist: /loja, /loja/*, /carrinho, /checkout, /minha-conta
  | { type: "start_checkout" }                            // storefront validates cart, then navigates to /checkout
  | { type: "handoff_human"; reason: string }
  | { type: "request_customer_info"; fields: Array<"cep" | "email" | "nome"> };
```

**Rules DivaHub must enforce:**

- Never emit `add_to_cart` for an inactive product or a variant with `stock = 0`.
  (DivaHub's product search already filters on `active=true` ŌĆö keep that invariant.)
- Never emit `apply_coupon` without knowing the code is valid. If uncertain,
  emit a `message` telling the user to try it in the cart page.
- `navigate` paths outside the whitelist will be **dropped** by the storefront
  and logged as a contract violation.
- Qty must be `1 <= qty <= 10` per action. Larger quantities ŌåÆ multiple actions
  or escalate to human.

## 8. Streaming protocol details

- Response `Content-Type: text/event-stream; charset=utf-8`.
- `Cache-Control: no-cache, no-transform`.
- `X-Accel-Buffering: no` (nginx passes this through; keeps chunks flowing).
- Each event terminated by `\n\n`.
- Send a heartbeat comment line `:\n\n` every 15 seconds while thinking, to keep
  proxies from closing the connection.
- Max turn duration: 60 seconds. After that, emit `error` with
  `code: "timeout"` and `turn.end`, then close.
- Storefront sets a 90-second client-side timeout.

**Non-streaming fallback (`Accept: application/json`)** must be implemented for
same-origin testing and for the admin "replay turn" tool. Behaviour is identical
except the response is a single JSON document after the LLM finishes.

## 9. Observability / logging contract

For every call, DivaHub must log (and expose in admin inbox):

- `turnId`, `conversationId`, `sessionKey`, `customerId?`
- `keyHint` (first 6 + last 3 chars ŌĆö never full token)
- `latencyMs`, `ttfbMs`, `tokensIn`, `tokensOut`
- `status` (200/401/422/429/503/ŌĆ”)
- Raw request body (PII allowed ŌĆö already PII-handling in DivaHub)
- `actions[]` emitted (so admins can audit what Divinha did on the site)
- `contractViolations[]` (storefront will POST these back if Divinha emits
  a malformed action ŌĆö see ┬¦10)

The storefront will mirror a thinner log in its own `IntegrationRun` table
(adapter: `divahub_divinha`, operation: `turn`). Conversation bodies live in
DivaHub only ŌĆö single source of truth.

## 10. Contract-violation reporting (reverse channel)

When the storefront receives an action it cannot execute (wrong type, invalid
SKU, stock gone, off-whitelist path), it **will drop the action** and POST a
short report to:

`POST /api/public/divinha/contract-violations`

```jsonc
{
  "turnId": "t_01H...",
  "conversationId": "cnv_01H...",
  "violations": [
    { "code": "unknown_variant_sku", "action": { "type": "add_to_cart", "variantSku": "BD-XXX", "qty": 1 } },
    { "code": "navigate_not_whitelisted", "action": { "type": "navigate", "path": "https://evil.com" } }
  ],
  "reportedAt": "2026-04-22T12:00:05Z"
}
```

DivaHub should surface these in the admin inbox for that conversation so
prompt/playbook regressions are caught quickly. Best-effort ŌĆö the storefront
fires and forgets.

## 11. Versioning

- URL prefix `/api/public/divinha/` is stable. Breaking changes bump the path
  to `/api/public/divinha/v2/`.
- Header `X-Brilho-Version: 1` identifies the caller's contract major.
- Additive changes (new action types, new message kinds, new optional fields)
  do **not** bump the major. The storefront must tolerate unknown action types
  by logging and dropping.
- When DivaHub ships a breaking change, run both versions in parallel for at
  least 2 weeks.

## 12. Guardrails ŌĆö Divinha behaviour requirements

These are product requirements, not schema. Must be enforced inside the LLM
orchestration, not just at the HTTP boundary:

1. **Language:** Always pt-BR. No fallback to en. Tone: warm, feminine, uses
   "voc├¬", avoids "tu". Matches DivaHub's existing Divinha persona.
2. **Prices/stock:** Never state a price or stock level. Refer to products by
   slug/SKU only. The storefront renders live prices.
3. **Promises:** Never promise delivery dates. Refer customers to checkout for
   live shipping quote (Correios).
4. **Discounts:** Never invent coupons. If the customer asks for a discount,
   either apply a coupon that exists (via `apply_coupon`) or escalate.
5. **Payment details:** Never ask for card numbers, CPF, or CVV inside the chat.
   Direct customers to `/checkout` via `start_checkout`.
6. **Refunds/complaints:** Escalate (`handoff_human`) ŌĆö do not commit the brand
   to any refund or replacement unilaterally.
7. **PII minimisation:** Only request CEP/email/name via `request_customer_info`
   when the flow genuinely needs it (e.g., shipping quote). Never ask for CPF
   or address in-chat.
8. **LGPD:** Honour a `/esquecer-conversa` trigger (storefront will POST
   `DELETE /api/public/divinha/conversations/:id` ŌĆö spec in v1.1). Until then,
   DivaHub retains conversations 24 months, then purges.
9. **Safe content:** Refuse NSFW, political, medical advice. Standard moderation.

## 13. Acceptance criteria ŌĆö DivaHub side

Before the storefront switches off its stub client and flips the Divinha widget
to live, the storefront team will run this checklist against a staging DivaHub
URL. All must pass.

- [ ] `GET /api/public/divinha/health` returns 200 + `ok: true` with a valid key.
- [ ] `GET /api/public/divinha/health` returns 200 + `ok: true` without a key (unauthenticated).
- [ ] `POST /api/public/divinha/turn` with no `Authorization` returns 401.
- [ ] `POST ŌĆ”/turn` with a revoked key returns 401.
- [ ] `POST ŌĆ”/turn` with a malformed body returns 422 + `issues[]`.
- [ ] `POST ŌĆ”/turn` with `Accept: text/event-stream` streams `turn.start`, Ōēź1 `token`, `turn.end` in order, within 60 s.
- [ ] `POST ŌĆ”/turn` with `Accept: application/json` returns a single JSON doc.
- [ ] `Idempotency-Key` replay with same body returns the same `turnId`.
- [ ] `Idempotency-Key` replay with different body returns 409.
- [ ] Heartbeat comments are sent every Ōēż15 s while thinking.
- [ ] 429 is returned with `Retry-After` header after the documented burst is exhausted.
- [ ] An `add_to_cart` action references a real `variantSku` from the catalog snapshot DivaHub received from the storefront.
- [ ] A conversation spanning 3 turns preserves `conversationId` and shows up in the DivaHub admin inbox with channel `storefront_web`.
- [ ] A turn that would require quoting a price instead references the product by slug and lets the storefront render the price.
- [ ] `POST ŌĆ”/contract-violations` accepts a report and surfaces it in the admin inbox for the relevant conversation.
- [ ] Logs show `keyHint`, never the full token.

## 15. Out of scope for v1 (flagged for later conversations)

- Voice input / TTS responses.
- Proactive messages (DivaHub pinging the user first). If added later, needs

- Returning product images/prices in the stream (storefront hydrates).
- File uploads (photos of jewellery to identify).
